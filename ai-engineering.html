ENGINEERING PHILOSOPHY

PHILOSOPHY

Everything I build started as a manual process first. These principles are just patterns I kept running into while doing the work.

🎯
Right Tool for the Job
I don't default to the most powerful model. Claude for reasoning and planning, DeepSeek or Qwen for code, cheaper API tiers where quality holds up. Matching the model to the task keeps costs down and output quality high.
🚀
Ship First, Iterate After
I'd rather have something live and imperfect than perfect and unfinished. Both ModEval and Outreach started as rough prototypes. The real learning happens once it's running against actual data.
🔧
Built From Experience
I build tools to solve problems I've actually lived. Not from a spec, not from a tutorial. That means I already know where things break, what edge cases matter, and what a good output actually looks like.
📡
If You Can't Debug It, It's Not Done
Every system I build has logging, clear outputs, and enough visibility to understand what happened when something goes wrong. Black boxes are a liability. I want to be able to look at any run and know exactly what the system did and why.
🤖
AI-Assisted Development
I'm not a traditional coder and I don't pretend to be. I use Claude to plan and architect, DeepSeek and Qwen via Aider for the actual code, and I pick the cheapest model that can do the job. What I ship is real and live. That's what matters.

AUTOMATION WORK

BUILDING WITH AI

I build automation pipelines that connect scraping, AI, and external APIs into end-to-end workflows. No manual steps, no babysitting. The job search pipeline is the clearest example of how I put this together.

🔍
Scheduled Scraping Pipelines
Playwright and BeautifulSoup running on a schedule, hitting target pages, parsing what matters, deduplicating against what's already been seen, and pushing clean data somewhere useful. Built this for job boards but the pattern works for anything.
🤖
AI in the Pipeline
I wire AI APIs into automation workflows to handle the steps that need actual reasoning. Summarizing, rewriting, classifying, extracting structure from unstructured text. The job search pipeline uses Claude for resume tailoring but the same pattern applies anywhere.
✍️
Browser Automation
Full browser automation for anything that needs actual interaction with a page. Form filling, multi-step flows, data extraction from pages that block simple scraping. Works where basic HTTP requests don't.
📊
Bringing It Together
The pipeline feeds a custom frontend where everything surfaces in one place. Scraping, AI processing, and a UI on top, all connected. That's the pattern I follow: automate the data flow, then build something to make it usable.
SYSTEM DESIGN

PRODUCTION VS TOY

Anyone can get something working locally. These are the things that separate a script from a system.

⚙️ What makes it production-ready
01
It Handles Failure
APIs go down. Scrapes return nothing. Models time out. Production systems expect this and keep running anyway. I build with fallbacks and failure isolation so one broken piece doesn't take down everything else.
02
It's Configurable
Hardcoded values are a liability. Thresholds, filters, targets -- these should be adjustable without touching the underlying logic. If changing one thing requires a code edit, the system isn't finished.
03
It Logs What It Does
If something goes wrong and you can't reconstruct what happened, you're guessing. Every system I build logs decisions, inputs, and outputs. Not for compliance, just because debugging blind is a waste of time.
04
It Stays Live Without Babysitting
Push to main, auto-deploy fires, secrets stay out of the repo. Both ModEval and Outreach run this way. Once it's configured, it just runs. That's the bar.
